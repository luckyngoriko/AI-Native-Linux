## DEC-001 — README "Self-Evolving Backend" reframed as "Adaptive Backend"

- **Context:** Rev.1 README contained a "Self-Evolving Backend" section that implied autonomous AI patching of backend code, in conflict with the spec discipline of typed actions, policy gating, and evidence-first execution. Rev.1 SPECIFICATION.md never sanctioned self-evolution.
- **Decision:** Reframe as "Adaptive Backend": AI may _propose_ backend patches, kernel adaptation adjustments, distro compatibility profiles, and new runtime adapters; production promotion always requires human approval. The Cognitive Core may not modify the Policy Kernel, the Evidence Log, the Vault Broker, or the recovery boot path through this pipeline.
- **Consequences:** README and SPECIFICATION are now consistent on the AI's bounded execution model. The pipeline (Observe → Propose → Sandbox simulation → Tests → Human review → Staged deployment → Monitor → Rollback) is fully explicit.
- **Status:** `REAL` (applied in initial commit `be318da`)
- **Phase tag:** S0.2

## DEC-002 — Repository layout: revision-per-folder, layer-per-subfolder

- **Context:** Rev.1 was a single `SPECIFICATION.md` file. As rev.2 grows, a single file becomes unwieldy. Comparable projects (NeuroCAD `000.000.Roadmap/NEUROCAD_UNIFIED_PLATFORM_SPEC_REV*`) use one folder per revision with topic-named files inside.
- **Decision:** For AIOS:
  - One folder per revision: `001.AI-OS.NET--SPECREV.1/`, `002.AI-OS.NET--SPECREV.2/`, …
  - Inside each rev.2+ folder, one folder per layer (L0–L10) plus a `XX_Cross_Cutting/` folder for contracts shared by multiple layers.
  - Inside each layer folder, files numbered `00_overview.md`, `01_<topic>.md`, `02_<topic>.md`, …
  - Rev.1 stays as a flat verbatim move (the original two files) since it pre-dates this convention.
- **Consequences:** Clean navigation, easy to grow (e.g., L4 will hold Policy Kernel, Vault Broker, and Identity Model as three separate sub-specs). Slightly more nesting than NeuroCAD's flat-files-per-revision approach, but justified by AIOS layers containing multiple distinct sub-systems each.
- **Status:** `REAL` (applied in this commit)
- **Phase tag:** infrastructure

## DEC-003 — Action Envelope + Lifecycle (S0.1) design choices

- **Context:** Rev.1 §13 sketched an action envelope and lifecycle but left major gaps: idempotency, causality, formal error model, lifecycle FSM precision, schema versioning, OpenTelemetry handling, sandbox profile binding, and dry-run modes. S0.1 closes these per scope option **B — Pragmatic+** (agreed at brainstorming).
- **Key decisions captured in [`XX_Cross_Cutting/01_action_envelope_lifecycle.md`](XX_Cross_Cutting/01_action_envelope_lifecycle.md):**
  - **Envelope style:** custom AIOS-native, proto-first, with `request`/`execution` field-level separation borrowed from Kubernetes API machinery. CloudEvents (event-style, not command-style) and full K8s API machinery (overkill for AIOS) were rejected.
  - **Top-level partition:** `schema_version`, `identity`, `request`, `execution`, `trace`. `identity` and `request` are caller-set and immutable after creation; `execution` is runtime-set and mutates over the lifecycle; `trace` is transport metadata.
  - **IDs:** ULID (Crockford base32, 26 chars), prefix-namespaced (`act_`, `intent_`, `plan_`, `corr_`, `evr_`, `polreq_`, `poldec_`, `appr_`). Chosen over UUID for chronological sortability and compactness.
  - **Idempotency:** Stripe-style separate `idempotency_key` field with 24h default TTL. Same key + same `hash(request)` deduplicates; same key + different hash returns `IdempotencyConflict`.
  - **Causality:** single-parent `parent_action_id` with cycle detection. Multi-parent saga deferred.
  - **Lifecycle:** five-phase FSM (`PENDING`, `RUNNING`, `SUCCEEDED`, `FAILED`, `ROLLED_BACK`) with K8s-style `conditions[]` for fine-grained facts. Six valid transitions; terminal phases are truly terminal (post-hoc rollback is a new envelope, not a transition).
  - **Result and Error:** two independent optional fields (not `oneof`), enabling `ROLLED_BACK` to populate both (cause + rollback summary). Population rules enforced by Capability Runtime.
  - **Error taxonomy:** ~30 canonical PascalCase codes grouped into validation/policy/auth/execution/verification/rollback/infrastructure. Recursive `cause` chain bounded at depth 8. `retryable` hint as a non-binding caller hint. English `message`; localization is a renderer concern.
  - **Versioning:** proto API style (`v1alpha1` → `v1betaN` → `v1` → `v2`), not SemVer. Stable allows additive changes only; breaking changes require major version bump with 12-month dual-version support.
  - **Canonical encoding:** proto3 deterministic serialization for the wire and idempotency hashes; JCS (RFC 8785) for JSON projection in evidence. **BLAKE3** (256-bit) for all canonical hashes.
  - **Trace context:** standard W3C Trace Context, no AIOS-specific extensions; child actions inherit parent `trace_id`.
  - **Sandbox profile:** resolved at execution start as `max_restriction(policy_required, caller_request, adapter_default)`. Fail-closed if all unset.
  - **Dry-run modes:** `LIVE` (default), `VALIDATE` (schema/idempotency only), `SIMULATE` (full path with simulated execution; sandbox still applied; evidence marked `simulated=true`). Adapters that handle destructive operations must declare a `simulate` capability.
  - **gRPC interface:** simplified from rev.1's nine RPCs to six. Single entry point `SubmitAction` (replaces `Validate`/`EvaluatePolicy`/`Execute`/`Verify`/`Rollback` orchestration). Streaming `WatchAction` replaces polling. New `GetCapabilityRuntimeInfo` for version negotiation.
- **Out of scope (intentionally deferred):** subject canonical format (L4 identity), saga composition, approval delivery mechanics (L4), TTL/expiration, resource budgets.
- **Consequences:** L3 (Capability Runtime), L4 (Policy Kernel client), L5 (Cognitive Core), L9 (Evidence) now have a stable contract to build against. Phase 1 sub-specs (Capability Translator S1.1, Latency Tiering S1.2, AIOS-FS object model S1.3) can proceed without re-litigating action contract questions.
- **Status:** `CONTRACT` (design approved 2026-05-07; awaits implementation evidence to escalate to `REAL`).
- **Phase tag:** S0.1
- **Schema version:** `aios.action.v1alpha1`

## DEC-004 — Capability Translator as catalog-bound compiler, not command generator

- **Context:** Rev.1 required that AI never directly execute shell commands and instead produce typed actions. S0.1 defined the action envelope, but L5 still needed a precise contract for translating natural-language intent and planner steps into those envelopes at large catalog scale.
- **Decision:** Define the Capability Translator as a catalog-bound compiler. It may use LLMs for parsing, extraction, explanation, and candidate suggestion, but every READY result must validate against a known capability manifest, the selected target schema, and the S0.1 envelope schema. Vector retrieval is allowed for candidate discovery but cannot be final authority. Ambiguity returns clarification; missing schema fields block translation; unknown actions and free-form shell execution are rejected.
- **Consequences:** AIOS gets a scalable path for thousands of typed capabilities without becoming a prompt-to-shell system. L3 adapter manifests become the source of truth for action availability. L4 remains the policy authority. L9 receives translation evidence for auditability. S1.2 can now specify latency routing between direct translation and cognitive translation without redefining translator semantics.
- **Status:** `CONTRACT` draft (awaiting review before design approval)
- **Phase tag:** S1.1

## DEC-021 — S8.2 L8 GPU Resource Model (initial contract — hardware peer of S7.1)

- **Context:** S7.1 Surface + Composition Model referenced `gpu_capability_class` as a closed enum to be defined in L8. S3.2 sandbox composition needed a typed GpuPolicy surface to bind to. Cross-group surface pixel-readback isolation (per S4.1 default-deny) needed an enforcement point at the GPU layer. Without an L8 GPU contract, all three were dangling. The user's 2026-05-10 architectural brainstorm explicitly called this out — Wave 2 ran the L7.2 schema and L8 GPU contracts in parallel via background agents.
- **Decision:** Write `L8_Network_Hardware_Devices/05_gpu_resource_model.md` (803 lines) as the typed GPU resource model. Numbered `05` after the four originally planned L8 sub-specs (`01_hardware_graph`, `02_network_policy`, `03_dns_vpn_management`, `04_firmware_trust`) so existing planned numbering remains intact.
  - **Closed `GpuCapabilityClass` enum (5 values)** with concrete VRAM defaults: `GPU_PASSIVE_DISPLAY` 16 MiB (no shader execution; AIOS chrome blitting), `GPU_BASIC_2D` 64 MiB (2D shaders, low queue priority), `GPU_RICH_2D` 256 MiB (full 2D + simple 3D), `GPU_FULL_3D` 25% of total VRAM minimum 256 MiB (full GPU access, large allocation), `GPU_COMPUTE_HEAVY` 50% of total VRAM minimum 512 MiB (GPGPU compute access for ML inference and simulation).
  - **Closed `QueuePriority` (NICE/NORMAL/HIGH), `GpuKind` (INTEGRATED/DISCRETE), `CapabilityBindingState` (REQUESTED → ACTIVE → EXPIRED|REVOKED).** Class-to-kind routing: GPU_FULL_3D and GPU_COMPUTE_HEAVY prefer DISCRETE; GPU_PASSIVE_DISPLAY and GPU_BASIC_2D prefer INTEGRATED unless policy specifies.
  - **Per-group VkDevice partitioning on native:** one VkDevice per active group on the host's primary GPU; secondary GPUs partitioned similarly. 25% per-group VRAM floor default. Inactive groups (no active sessions) lose their VkDevice after `inactive_group_vk_device_ttl` (default 5 minutes); resources reclaimed and `GPU_DEVICE_FORCE_RECLAIMED` evidence (FOREVER) emitted. KWin reserve: 256 MiB hard cap separate from group budgets.
  - **Per-origin GPUAdapter sandboxing on Web:** browser iframe enforcement; AIOS web renderer sets correct iframe origin per group so that GPUAdapter cannot leak across groups even if the spec layer were forgiving (which it is not).
  - **Cross-group dmabuf passing denied by default (`DmabufCrossGroupForbidden`)** — matches S4.1 cross-group default-deny invariant. The single privileged peer is `_system:kwin` (the KDE compositor): constitutionally allowed to receive dmabuf from any group surface for composition purposes (composes opaquely without copying pixels into another group's VkDevice). Per-class peer sets:
    - `GPU_PASSIVE_DISPLAY` → kwin only
    - `GPU_BASIC_2D` and `GPU_FULL_3D` → same `(group, subject)` + kwin
    - `GPU_RICH_2D` → same group + kwin
    - `GPU_COMPUTE_HEAVY` → same `(group, subject)` only (no compositor; compute results never reach a screen surface unless explicitly converted to render output via a separate render-class binding).
  - **dmabuf revocation:** fence-counter increment on revoke; pre-revocation kernel mappings persist until process exit; force-teardown available via `RevokeCapabilityBinding(destroy_underlying_vk_device=true)` for emergency reclaim. Stale dmabuf handles after revoke fail at use.
  - **Hardware capability-lie detection at every binding request.** Driver claims one ABI version; kernel exposes another. Re-probe at binding time; mismatch → `HOST_CAPABILITY_LIE` evidence (FOREVER) + binding refused. Mirrors S3.2 §6.3 capability-lie discipline.
  - **IOMMU-unavailable degraded mode.** Without IOMMU, cross-group VkDevice memory isolation degrades. AIOS continues operation but emits `IOMMU_UNAVAILABLE_DEGRADED` evidence (EXTENDED_60M) and operator-visible warning; cross-group surface pixel-readback denial still attempted at the device-level Vulkan layer but contract is best-effort, not constitutional, when IOMMU is missing.
  - **Recovery-mode disables Vulkan validation layers** to reduce attack surface; emits `GPU_VALIDATION_DISABLED_RECOVERY` evidence. On normal-mode boot validation is re-enabled with `GPU_VALIDATION_ENABLED_NORMAL` evidence.
  - **Eight golden fixtures:** single-group GPU usage with capability binding; cross-group VkDevice isolation (family + homelab surfaces on same display, separate VkDevices, no shared memory); cross-group dmabuf denial; VRAM exhaustion fallback (downgrade GPU_FULL_3D request to GPU_RICH_2D with `GPU_BUDGET_DOWNGRADED` evidence); hardware capability lie detected (driver claims Vulkan 1.3, kernel reports 1.1); compositor privileged dmabuf access (KWin in `_system` scope receives all groups' dmabufs for composition); recovery-mode validation disabled; multi-GPU routing (compute → discrete; passive → integrated).
  - **Twelve metrics with bounded label cardinality:** subject/surface/device/group ids NEVER as labels; `capability_class`/`result`/`error_code`/`gpu_kind`/`priority` are the only labels (all closed enums).
  - **Sixteen new evidence record types queued for S3.1 vocabulary follow-up:** ten STANDARD_24M (`GPU_DEVICE_ENUMERATED`, `GPU_DEVICE_DISCONNECTED`, `GPU_VK_DEVICE_CREATED`, `GPU_VK_DEVICE_DESTROYED`, `GPU_DMABUF_GRANTED`, `GPU_DMABUF_DENIED`, `GPU_CAPABILITY_DENIED`, `GPU_VALIDATION_DISABLED_RECOVERY`, `GPU_VALIDATION_ENABLED_NORMAL`, `DRIVER_UNAVAILABLE`); three EXTENDED_60M (`GPU_BUDGET_EXCEEDED`, `GPU_BUDGET_DOWNGRADED`, `IOMMU_UNAVAILABLE_DEGRADED`); three FOREVER (`HOST_CAPABILITY_LIE`, `GPU_BINDING_FORGERY`, `GPU_DEVICE_FORCE_RECLAIMED`).
  - **Other cross-spec touch-ups queued for follow-up:** S2.3 new closed condition fields `target.gpu_capability_class` + `target.gpu_device_kind` plus a hard-deny candidate `GpuComputeOutsideAuthorisedClass`; S2.4 new verification primitive `gpu_binding_class()`; S3.2 new field `SandboxProfile.gpu_policy: GpuPolicy`; S2.1 new query field `target.gpu_capability_class`.
- **Consequences:** L8 layer transitions from `SHELL` to `PARTIAL`. The dangling `gpu_capability_class` reference from S7.1 is now closed. S3.2's GpuPolicy sandbox field has a concrete enum surface to bind to. Cross-group GPU isolation has an enforcement point (per-group VkDevice on native, per-origin GPUAdapter on Web; dmabuf brokering with cross-group denial). The AIOS rendering architecture (DOM + WebGPU on Web; Qt + wgpu on KDE) now has matched contracts at L7 (Surface + UI schema) and L8 (GPU resource). Cross-spec touch-ups across S2.1, S2.3, S2.4, S3.1, S3.2 and L0 PropertyType extension are queued; the touch-up consolidation cycle will apply them all together.
- **Status:** `REAL` (applied 2026-05-10).
- **Phase tag:** S8.2.

## DEC-020 — S7.2 L7 Shared UI Schema (initial contract — render-target-agnostic node hierarchy)

- **Context:** S7.1 (Surface + Composition Model) fixed the GPU-canvas substrate but every renderer still needs a typed structural grammar to compile to its target primitives (DOM on Web, Qt/QML on KDE, terminal text on CLI). Without this, each renderer would invent its own internal node types and INV-019 (visual identity preserved across renderers) would be unenforceable at the structural layer. The user's 2026-05-10 hybrid rendering decision (DOM + WebGPU on Web; Qt + wgpu on KDE) requires that the same UI tree compile cleanly to both targets — that is the shared schema's job.
- **Decision:** Write `L7_Interaction_Renderers/02_shared_ui_schema.md` (1037 lines) as the structural grammar consumed by every renderer. Visual treatment (colors, typography, spacing, motion, AI-vs-human visual distinction per INV-021, recovery aesthetic per INV-022) is intentionally NOT in this spec; it lives in L7.3 Visual Language to be written next. The schema only declares the flags (`is_ai_origin`, `recovery_only`, `is_trust_bearing`) that the visual language consumes.
  - **Closed `NodeKind` enum (19 values):** TEXT, LIST, FORM, CARD, TABLE, VISUALIZATION, STREAM, SURFACE_EMBED, CONTAINER, SECURITY_INDICATOR, APPROVAL_PROMPT, EVIDENCE_LINK, AGENT_MESSAGE, ACTION_BUTTON, DIVIDER, SPACER, HEADING, INLINE_CODE, CODE_BLOCK. Adding a kind requires a versioned spec change. Renderers reject trees containing unknown kinds. `CONTAINER` was added on top of the 18 originally suggested because layout-only grouping needs a payload-free node distinct from `CARD` (`CARD` carries title + body + actions; `CONTAINER` carries only layout hints).
  - **Closed `LayoutMode` (7) + `SizeMode` (4):** STACK_VERTICAL, STACK_HORIZONTAL, FLOW_WRAP, GRID, SCROLL_VERTICAL, SCROLL_HORIZONTAL, FIXED. SizeMode: HUG_CONTENT, FILL_PARENT, FIXED_LOGICAL_PIXELS, FRACTION_OF_PARENT. No CSS, no floats, no absolute positioning at schema layer (renderer does that as needed for chrome zones via S7.1 composition).
  - **Closed `AccessibilityRole` (18):** WCAG-aligned roles (button, link, list, listitem, heading, banner, navigation, main, complementary, dialog, alert, status, region, form, group, separator, marquee, tooltip). Every node carries `accessibility_label` and optional `accessibility_role`; renderer enforces a11y at compile time.
  - **Closed `FormFieldKind` (7):** TEXT, MULTILINE, NUMBER, EMAIL, PASSWORD, CHECKBOX, SELECT. Keeps the form-input vocabulary closed.
  - **Closed `ApprovalChoice` (3):** ACCEPT, REJECT, DEFER. DEFER added beyond the suggested ACCEPT/REJECT to support "I need to think about this" without forcing rejection — reduces approval-fatigue-driven false rejections.
  - **Trust-bearing nodes constitutional:** SECURITY_INDICATOR, APPROVAL_PROMPT, EVIDENCE_LINK cannot be authored by AI subjects. The tree-signing service refuses to sign a tree containing those kinds when the issuer's `SubjectKind = AI_AGENT`. This binds INV-020 (trust indicators always visible) at the authorship layer — an AI cannot smuggle in a fake security indicator. AGENT_MESSAGE payload is validated to forbid trust-bearing descendants — prevents AI from nesting fake indicators inside its own messages.
  - **`is_ai_origin` is signer-controlled, not author-controlled:** the flag is set by the tree-signing service from the issuer's identity record, not from the issuer's input. AGENT_MESSAGE always carries `is_ai_origin = true`. Renderer applies INV-021 distinct visual treatment to every node with this flag (the visual language defines what "distinct" means). Forging the flag is impossible — the tree's signature won't validate if the flag is altered.
  - **`recovery_only` flag with two-mode handling:** outside recovery, a tree containing `recovery_only` nodes is rejected with `RecoveryNodeOutsideRecovery` if those nodes are required for the tree to make sense; if they are dropped silently, `recovery_node_dropped_total` telemetry is incremented. The discipline binds INV-022 at the authorship layer — recovery branches cannot leak into normal-mode rendering.
  - **Subtree Ed25519 signature:** the issuer signs the canonical encoding (JCS) of the tree. Renderer verifies before compilation; tampered trees produce no partial render. Identity-bearing fields (`is_ai_origin`, `is_trust_bearing`, `recovery_session_required`) are part of the signed payload — tampering invalidates signature.
  - **Stable `node_id = uin_<ulid>`:** identity preserved across re-renders triggered by data-binding updates. Renderer reuses node_id to keep focus, scroll position, and accessibility tree continuity. Within one tree, node_ids are unique; collision rejected at validation.
  - **Data binding to S2.1 query views:** LIST and TABLE nodes optionally carry `view_ref`; renderer subscribes to the view and re-renders on changes. Cardinality bound is 10 000 items per view (consistent with S2.1 §11.3 inbox bounds); larger views require cursor pagination.
  - **FORM submission canonical:** a submitted FORM produces an S0.1 action envelope where `request.body` is the form's field map and `submitter` is the active session subject — NEVER the tree's issuer when they differ. This separates accountability (who chose to submit) from authorship (who designed the UI). Same for ACTION_BUTTON.
  - **Per-tree 10 000-node bound:** prevents schema-flood DoS. Trees exceeding the limit rejected with `SchemaTreeTooLarge`.
  - **Schema versioning:** same proto v1alpha1 / v1betaN / v1 / v2 discipline as S0.1. Adding a NodeKind = breaking change; renderers declare supported version and reject newer.
  - **Eight golden fixtures, twelve metrics with bounded cardinality** (kind/node_count_class/result/error_code/recovery_mode/is_ai_origin as labels; node_id/subject_id/group_id NEVER as labels).
- **Consequences:** L7 layer remains `PARTIAL`; foundation pair S7.1 + S7.2 are now both `CONTRACT`; downstream sub-specs (`03_visual_language`, `04_kde_renderer`, `05_web_renderer`, `06_cli_renderer`, `07_voice_mobile_future`) consume both. Three new evidence record types queued for S3.1 follow-up: `UI_TREE_VALIDATION_REJECTED` STANDARD_24M, `UI_TRUST_BEARING_AUTHORSHIP_REFUSED` FOREVER, `UI_RECOVERY_NODE_DROPPED` STANDARD_24M. Two new S2.4 verification primitives queued: `tree_contains_kind(tree_id, kind)` and `tree_max_depth(tree_id)`. The schema is target-agnostic by construction — KDE renderer compiles `Node{kind=FORM, ...}` to `QFormLayout`; Web renderer compiles same node to `<form>` DOM; CLI renderer compiles to ANSI-bordered prompt sequence. Rendering target is the renderer's responsibility; the schema itself stays unaware.
- **Status:** `REAL` (applied 2026-05-10).
- **Phase tag:** S7.2.

## DEC-019 — S7.1 L7 Surface + Composition Model (initial contract — foundation under all renderers)

- **Context:** The L7 layer was a `SHELL` overview with five planned renderer-specific sub-specs (KDE, Web, CLI, Voice, Mobile). The user's 2026-05-10 brainstorm for AIOS rendering architecture (DOM + WebGPU hybrid for Web; Qt/QML + wgpu for KDE; supporting games and CAD as first-class app surfaces) revealed that ALL renderers needed a common foundation — a typed Surface primitive with composition rules, capability brokering, cross-group isolation, AIOS chrome enforcement, and recovery-mode constraints. Without this foundation, each renderer would re-invent surface semantics and INV-019..INV-022 would be unenforceable. This contract creates the foundation as a NEW sub-spec at the head of L7's planned list.
- **Decision:** Write `L7_Interaction_Renderers/01_surface_composition.md` (723 lines) as the L7 foundation. Renumber the existing planned sub-specs to make room: shared UI schema becomes `02`, visual language is added as `03` (was unplanned), KDE renderer becomes `04`, Web renderer becomes `05`, CLI renderer becomes `06`, voice/mobile becomes `07`. The contract:
  - **Closed `SurfaceKind` enum (4 values).** `AIOS_SURFACE` rendered by L7's UI schema engine; `APP_SURFACE` for apps that ship their own renderers (games via Bevy/Wine/Unreal, CAD via native Vulkan or wgpu, scientific viz, etc.); `STREAM_SURFACE` for live media (webcam, screen capture, evidence playback); `XR_SURFACE` reserved-but-deferred so future VR/AR additions don't cause a versioned schema break for current consumers. Adding a kind requires a versioned spec change.
  - **Closed `CompositionZone` enum (4 values) with strict z-ordering.** `BACKGROUND` (wallpaper, ambient — `AIOS_SURFACE` only) < `CONTENT` (main interaction — all kinds except `XR_SURFACE`) < `OVERLAY` (notifications, agent messages — `AIOS_SURFACE` only) < `CHROME` (security indicator, action context, recovery banner — `AIOS_SURFACE` only). Per-zone allowed-kind matrix is constitutional; CHROME exclusivity for `AIOS_SURFACE` directly implements INV-020 (trust indicators always visible).
  - **Five-state Surface lifecycle FSM** (`DRAFT → ACTIVE → PAUSED → RETIRED → DESTROYED`) with explicitly allowed transitions and rejected backward transitions. Surfaces are identity-bearing records; backward state would imply mutation which AIOS forbids on identity records.
  - **Capability-brokered GPU access.** Every Surface that uses GPU primitives carries a `gpu_capability_class` reference (closed enum to be defined in L8 GPU resource model — concurrent contract): `GPU_PASSIVE_DISPLAY` (no shader execution), `GPU_BASIC_2D`, `GPU_RICH_2D`, `GPU_FULL_3D`, `GPU_COMPUTE_HEAVY`. The renderer obtains a `GpuCapabilityBinding` from L8 service for the requesting subject + group + class triple; cross-group surface pixel-readback isolation enforced at the GPU layer (per-group `VkDevice` on native, per-origin `GPUAdapter` on Web). dmabuf passing across group boundaries is denied at L8.
  - **AIOS chrome is unbreakable from app surfaces.** Apps that submit to CHROME zone are rejected with `CompositionZoneForbidden` and emit `CROSS_ZONE_VIOLATION_ATTEMPTED` evidence. Fullscreen mode on app surfaces does not promote them above OVERLAY or CHROME — KDE enforces via `wlr-layer-shell` `overlay` layer which survives fullscreen; Web enforces via shadow-root chrome at z-index 9999 above page fullscreen. Implements INV-020 directly.
  - **Recovery-mode surface stack constraint.** When `subject.recovery_mode = true`, `APP_SURFACE` and `STREAM_SURFACE` creation is rejected with `RecoveryModeKindForbidden`; `RECOVERY_KIND_REJECTED` evidence (FOREVER retention) is emitted. Recovery shell is exclusively `AIOS_SURFACE` rendered with the recovery aesthetic. Implements INV-022 directly.
  - **Surface-id signing.** Surface records are signed by the renderer with an Ed25519 key issued by the identity service; consumers (compositor, evidence log, S0.1 envelope acceptance) verify signatures. Forged surface ids are rejected; surface impersonation is structurally impossible.
  - **Per-subject quota (default 32 active surfaces).** Prevents surface-flood DoS. Quota is per-subject not per-group, so one subject cannot starve peers in their own group.
  - **Cross-group composition adjacency without read.** Surfaces from different groups can sit on the same display visually but cannot read each other's pixels (L8 GPU isolation). The compositor knows each surface's group via `ScopeBinding` (S1.3 §21); pixel-readback attempts across boundaries fail at the device-isolation layer.
  - **`wgpu` unification across targets.** Apps written in Rust + `wgpu` compile once: the same source produces a native Linux binary using Vulkan via `wgpu` (KDE renderer hosts it via Wayland subsurface) AND a wasm binary using WebGPU via `wgpu` (Web renderer hosts it via `<canvas>`). The Surface contract is identical on both targets. Apps with non-portable renderers (Wine for DirectX games, Unreal/Unity native) are KDE-only; the schema marks `Surface{portable=false}` and the Web renderer shows a degraded placeholder.
  - **Eight golden fixtures.** AIOS evidence viewer surface; Bevy game on KDE with full GPU and CHROME-above-fullscreen; NeuroCAD on KDE and Web demonstrating `wgpu` unification; cross-group pixel readback denied at GPU layer; APP_SURFACE rejected in recovery mode; chrome remains above fullscreen game; surface-id forgery rejected; surface-flood quota prevents DoS.
  - **Seven new evidence record types added to S3.1 vocabulary.** `SURFACE_CREATED` STANDARD_24M, `SURFACE_DESTROYED` STANDARD_24M, `SURFACE_GPU_BUDGET_EXCEEDED` EXTENDED_60M, `CROSS_SURFACE_READ_DENIED` FOREVER, `CROSS_ZONE_VIOLATION_ATTEMPTED` EXTENDED_60M, `RECOVERY_KIND_REJECTED` FOREVER, `SURFACE_NEVER_RENDERED` STANDARD_24M.

## DEC-016 — S5.1 L4 Identity Model (initial contract)

- **Context:** S2.3 §3 (subject normalization) and S3.2 §5.2 (floor selection per `is_ai`/`is_recovery_mode`) referenced an "L4 identity" placeholder. S4.1 §12.8 added concrete constraints: subject canonical id format `<group_id>:<sub_id>`, group as first-class identity, single `primary_group_id` semantics, recovery-mode subjects always under `_system` scope. With those constraints in place L4 identity could be written rigorously rather than as a free-form sketch.
- **Decision:** Write `L4_Policy_Identity_Vault/03_identity_model.md` as a contract spec covering only the identity model (vault broker, approval mechanics, and emergency override remain in their own deferred sub-specs). The contract closes the seven-kind subject taxonomy, fixes the canonical id format, makes group a first-class unit, formalizes multi-group membership with single primary_group_id per session, separates recovery mode constitutionally, defines session class with per-class TTL budgets, and produces the `NormalizedSubject` that S2.3 §3 consumes.
  - **Subject taxonomy (closed):** seven kinds — `HUMAN_USER`, `AI_AGENT`, `APPLICATION`, `SERVICE`, `DEVICE`, `WORKFLOW`, `REMOTE_OPERATOR`. Adding a kind requires a versioned spec change. Each kind has a registration mechanism, default `is_ai`, and home location in the namespace.
  - **Canonical id:** `<group_part>:<kind_segment>[:<inner>]` with `group_part` being either a `group_id` or `_system`. Examples: `family:alice` (HUMAN_USER), `family:family-assistant` (AI_AGENT, group-owned), `family:alice:diary-helper` (AI_AGENT, personal under alice), `_system:service:translator` (SERVICE), `family:alice:device:thinkpad-x1` (DEVICE), `homelab:app:bg.iconys.proxguard:i-01` (APPLICATION), `finance:workflow:quarterly-close:run-7842` (WORKFLOW), `_system:remote:operator-247` (REMOTE_OPERATOR). Total length cap 256 bytes; kind-specific inner regex tighter than the loose pattern.
  - **Identity is unforgeable:** every action carries `Subject` and `NormalizedSubject` records signed by the identity service with Ed25519. The Capability Runtime rejects envelopes with bad signatures. `is_ai` is bound at registration and signed; subjects cannot tamper.
  - **Group as first-class unit:** `Group` message with `tier` (closed enum `PERSONAL`/`TEAM`/`ORGANIZATIONAL`/`FINANCE` — informative, maps to S3.2 group floor bundle). Group registration is a recovery-mode action that emits `GROUP_REGISTERED` evidence with FOREVER retention. No nested groups in Rev.2.
  - **Multi-group membership with single primary:** a subject's `Memberships` list can contain multiple groups, but exactly one is `is_primary = true` per session. Switching primary requires re-authentication (`SwitchPrimaryGroup` RPC closes the old session and issues a new one with a new canonical id like `homelab:alice` instead of `family:alice`); emits `PRIMARY_GROUP_SWITCHED` evidence. Capabilities bound under one membership are not active under another.
  - **Recovery mode constitutionally separate:** recovery subjects always under `_system` scope (`_system:remote:operator-<id>`, `_system:local:operator-<id>`). No path from a normal-mode session to a recovery-mode session except by reboot. 8-hour hard cap on recovery sessions; no extension. AI subjects rejected from `RECOVERY` session class (`AISubjectCannotEnterRecovery`).
  - **Session class (closed):** `PUBLIC` / `INTERACTIVE` / `STRONG` / `RECOVERY` / `SERVICE`, each with default and hard-maximum TTL. Re-authentication produces a new session id; sessions are not refreshed in place.
  - **Risk flags (closed):** seven kinds including `ELEVATED_PRIVILEGE_REQUEST` automatically set on AI subjects acting outside their home scope (group agent acting in `users/<u>/`, personal agent acting in `shared/`). Enables S2.3 to require approval for any AI action that escapes the agent's home scope.
  - **`NormalizedSubject` for S2.3:** carries `canonical_subject_id`, `kind`, `is_ai`, `primary_group_id`, `memberships`, `capabilities` active in this session, `session_class`, `recovery_mode`, `risk_flags`, `identity_bundle_version`, plus Ed25519 signature. Policy kernel verifies signature on every evaluation.
  - **Capability binding skeleton:** `CapabilityBinding` scoped to `(subject_canonical_id, group_id, identity_bundle_version)`. Bundle version rollover invalidates bindings — vault broker re-validates per active version. This bounds the impact of a leaked binding to one bundle version's lifetime. Full vault broker API deferred to `02_vault_broker.md`.
  - **Identity bundle:** signed by AIOS root, content-addressed `idbundle_<hex_lower(BLAKE3(...))[:32]>`. Signature failure puts identity service in degraded mode (only constitutional `_system` subjects available). Bundle change applies to new sessions only; existing sessions continue under their authentication-time bundle until expiry.
  - **AI subject discipline:** `is_ai = true` only for subjects with `SubjectKind = AI_AGENT`. Apps using LLMs internally do not become AI subjects. Combined with S2.3 §17 self-approval prevention and §26.2.3 `AISystemAdminBlocked`, AI subjects have no path to system mutation. AI agent retirement and replacement always produces a new canonical id even if the `agent_id` segment is unchanged.
  - **Determinism contract:** `(canonical_subject_id, primary_group_id, identity_bundle_version, vbset_version, session_class, recovery_mode) → identical NormalizedSubject`. Cacheable per the same tuple; cache invalidates on bundle or vault binding set version change.
  - **Performance budgets:** `AuthenticateSubject` p95 < 200 ms; `ResolveSubject` cached p99 < 2 ms / fresh p99 < 20 ms; `EnumerateMemberships` p99 < 5 ms; `SwitchPrimaryGroup` p95 < 500 ms; `IssueCapabilityBinding` p95 < 200 ms. All failure modes fail closed.
  - **Adversarial robustness:** forged subject blocked by signature verification; replay protection via per-authentication unique session_id; group-id collision resolved by S1.3 optimistic concurrency; capability binding leak bounded to one bundle version; AI impersonating human blocked at session issuance; recovery escalation impossible without reboot; membership enumeration leak prevented (RPC returns only the calling subject's memberships).
  - **Cross-spec dependencies:** S0.1 (subject + signature in envelope), S2.1 (queryable subject fields), S2.3 (NormalizedSubject consumed by §3), S2.4 (`POLICY_AI_SELF_APPROVAL_BLOCKED` property), S3.1 (records carry `subject_canonical_id`; new record types `GROUP_REGISTERED` FOREVER + `PRIMARY_GROUP_SWITCHED` STANDARD_24M), S3.2 (`is_ai` and `recovery_mode` drive floor selection), S4.1 (canonical id format), plus deferred S6.1 vault broker / S6.2 approval mechanics / S6.3 emergency override constraints.
  - **Six golden fixtures:** personal household authentication (alice + family-assistant), multi-group primary switch (alice in personal + work-finance), recovery operator session with 8-hour cap, AI agent rejected from recovery, forged subject signature rejection (new `SUBJECT_SIGNATURE_FAILURE` record type added to S3.1 vocabulary), capability binding scope enforcement across primary group switch.
  - **Telemetry:** eleven metrics with bounded cardinality (subject/group/user/session ids NEVER as labels); kind/result/error_code/cache/session_class/is_one_shot/reason/risk_flag_kind as the only label set.
- **Consequences:** L4 identity moves from `SHELL` to `CONTRACT` (S5.1). S2.3 §3 subject normalization and S3.2 §5.2 floor selection now have a real contract to bind to instead of a placeholder. Three new evidence record types added to S3.1 vocabulary (`GROUP_REGISTERED` FOREVER, `PRIMARY_GROUP_SWITCHED` STANDARD_24M, `SUBJECT_SIGNATURE_FAILURE` FOREVER) — total now 27. The vault broker, approval mechanics, and emergency override sub-specs remain deferred but their interfaces with identity are now constrained: capability binding scoped to `(subject, group_id, bundle_version)`, approvals bound to `subject_canonical_id` + `request_hash`, overrides require `kind = HUMAN_USER` + `recovery_mode = false` + STRONG session class.
- **Status:** `REAL` (applied 2026-05-09).
- **Phase tag:** S5.1.

  - **D2 — Proto IDL + gRPC service:** `aios.fs.v1alpha1` package; `AIOSFSObjects` service with `BeginTransaction`/`WriteVersion`/`PromotePointer`/`CommitTransaction`/`AbortTransaction`/`ReadObject`/`ReadVersion`/`ReadChunk`/`EnumerateObjects`/`ListConflicts`/`ResolveConflict`/`RebuildIndexes`/`QuarantineVersion`/`ExitQuarantine`/`RetireObject`/`PurgeObject`. Appendix A in object_model holds full IDL; Appendix B in conflict_resolution adds conflict-specific records.
  - **D3 — Transaction model with multi-pointer atomicity:** transactions can write multiple versions and promote multiple pointers; all CAS succeed or all fail (two-phase commit fence within one AIOS-FS instance).
  - **D4 — Chunking strategy + GC contract:** FastCDC defaults (min=64KB, avg=256KB, max=1MB); fixed-size permitted as fallback for streaming workloads; orphan staging TTL 24h; reference counting via active version refs; GC passes are evidence-logged, never silent.
  - **D5 — Privacy class and object lifecycle:** `PrivacyClass` field on objects mirrors the S1.2 §5 enum (`PUBLIC`/`INTERNAL`/`SENSITIVE`/`SECRET_BEARING`/`CLASSIFIED`); class can be raised but not lowered. Object lifecycle adds `ACTIVE`/`RETIRED`/`PURGED` with default 90-day retention; shortening retention requires policy approval.
  - **D6 — Pointer move CAS protocol:** atomic compare-and-swap on `(pointer_id, expected_current_version_id)`; standardized `ConflictDetected` error gateway to the conflict resolution sub-spec; multi-pointer transactions emit one conflict per failed pointer.
  - **D7 — Read consistency model:** `SNAPSHOT` (default; consistent across pointers as of read time), `LINEARIZABLE` (latest committed; for synchronization), `EVENTUAL` (for views; may lag).
  - **D8 — Quarantine semantics:** triggers (validation/integrity/policy/external attestation/operator), effects (pointer rollback, restricted reads), exits (manual review, automated re-validation), 30-day quarantine TTL leading to `RETIRED_VERSION`, evidence trail.
  - **D9 — CRDT vocabulary:** closed set `G_COUNTER`/`PN_COUNTER`/`OR_SET`/`LWW_REGISTER`/`OR_MAP`/`RGA_TEXT`; per-object-kind merge policy declares which CRDT applies; new types require additive enum bump.
  - **D10 — Conflict timeout, notification, authority:** 30-day default TTL → `ABANDONED` status with evidence; push notifications via L9 evidence stream with debouncing (5s coalescing window); pull via `ListConflicts`; resolution authority is owner / collaborator / operator-with-override, evaluated by L4 Policy Kernel.
  - **D11 — AI merge proposal validation:** structural rejection rules in code (proposal must include base/current/candidate/resolution/explanation/verification/evidence; rejected on empty verification, missing evidence, secret-shaped explanation per S1.1 §17.2.6, or AI involvement at privacy class beyond merge policy allowance); auto-promote requires `PUBLIC`/`INTERNAL` class plus explicit policy opt-in.
  - **D12 — Golden fixtures and telemetry contracts:** seven fixtures for object_model (write-promote, CAS conflict, multi-pointer atomicity, quarantine, GC, privacy monotonicity, recovery enumerate, snapshot read); seven fixtures for conflict_resolution (default reject, CRDT auto-merge, AI redaction, multi-pointer conflicts, TTL abandon, unauthorized resolve, resolution race). Telemetry metrics with bounded label cardinality; subject never a metric label.
- **Consequences:** S1.3 grows from 264 to 1428 lines combined (object_model 913, conflict_resolution 515). Object model becomes the authoritative source for the L2 gRPC surface; conflict resolution operates on the object model's CAS protocol. Multi-pointer transactions are now contract-grade. AI auto-promote conditions are explicit and verifiable.
- **Status:** `REAL` (applied 2026-05-08).
- **Phase tag:** S1.3 refinement.

## DEC-008 — S1.2 Latency Tiering refinement (deltas D1–D12)

- **Context:** S1.2 landed as a 157-line draft in commit `dfa3be5`. Architectural shape was correct (5 tiers, T0/T1 must work without external AI, no tier bypasses safety) but the contract was tilt-heavy on bullet lists and missing typed surfaces, numbers, and adversarial defense.
- **Decision:** Apply twelve deltas without scope expansion:
  - **D1 — Proto IDL + `LatencyRouter` service:** typed `RoutingRequest`, `RoutingDecision`, `RoutingResponse`, `RoutingEvidence`, `RouterInfo`. Stable identifier `rt_<ULID>`. `Route()` and `GetRouterInfo()` RPCs. Full IDL in Appendix A.
  - **D2 — Quantitative per-tier budgets:** T0 p95 < 10 ms, T1 < 50 ms, T2 < 200 ms, T3 < 800 ms, T4 < 3 s; hard timeouts; cold start budgets. Aligned with S1.1 §19; this section is now authoritative.
  - **D3 — Deterministic routing algorithm:** priority-ordered guards (recovery mode → privacy class → policy egress → budget → rate limit) followed by tier selection (T0 cache → exact match → lexical → ambiguity → multi-step → default). Lowest tier wins on tie.
  - **D4 — `PrivacyClass` enum:** closed set `PUBLIC`, `INTERNAL`, `SENSITIVE`, `SECRET_BEARING`, `CLASSIFIED` with per-class tier ceiling table. Default `SENSITIVE`; secret detection raises to `SECRET_BEARING`; class can be raised but never lowered.
  - **D5 — Tier dynamics:** auto-upgrade rules (T1→T2 on miss, T2→T3 on low confidence + bind fail, T3→T4 on multi-step) capped at one upgrade per request; auto-downgrade on tier unavailable; hard timeout returns `TIMEOUT` outcome (never silent fallback).
  - **D6 — Evidence chain:** `routing_id → translation_id → action_id → evidence_receipt_ids`. `RoutingEvidence.translation_ids[]` filled when downstream translations produced. `correlation_id` flows through all four levels.
  - **D7 — Adversarial protection + cost model:** per-subject rate limits (60 routing/min, 20 T3/min, 10 T4/hour, 200 external/day defaults); external model budget guards with 80%/95%/100% threshold actions; defense delegation table.
  - **D8 — T0 caching semantics:** explicit cacheable / never-cacheable rules; cache key formula `rtc_<hex_lower(BLAKE3(JCS(...)))[:32]>`; TTL defaults (60 s status, 5 m views); invalidation triggers (catalog flip, evidence-event, session expiry, operator flush).
  - **D9 — Telemetry contract:** Prometheus-compatible metrics with bounded label cardinality (subject is never a label); histogram buckets for latency; required counters and gauges enumerated.
  - **D10 — Golden routing fixtures:** ten `{ input, expected, status }` triples covering exact, fuzzy, ambiguous, multi-step, recovery, secret-bearing, budget-exhausted, Bulgarian, adversarial-rate-limit, and classified-context cases.
  - **D11 — Statelessness contract:** router stateless across `Route()` calls (mirrors S1.1 §18 discipline); reproducibility input set documented.
  - **D12 — User preferences:** `RoutingPreference` with `prefer_local`, `prefer_speed`, `prefer_quality` as hints; policy and guards always override preferences; preferences are per-call, not persisted by router.
- **Consequences:** S1.2 grows from 157 to 933 lines. Becomes authoritative for latency budgets (S1.1 §19 was already deferring). New canonical refusal paths via `RoutingOutcome.REFUSED` and `RoutingOutcome.TIMEOUT`.
- **Status:** `REAL` (applied 2026-05-08).
- **Phase tag:** S1.2 refinement.

## DEC-007 — S1.1 Capability Translator refinement (deltas D1–D9)

- **Context:** S1.1 landed as a draft in commit `dfa3be5` together with the rest of the rev.2 roadmap (DEC-005). The draft was structurally sound but missing several contract-grade pieces.

  - **D2 — Adversarial robustness section:** explicit threat model (prompt injection, secret smuggling, ambiguity exploitation, action aliasing, embedded shell, social engineering of `reason`); structural defenses (catalog-only action names, schema-only target fields, no shell surface, deterministic identity, sanitized `reason`, secret-shaped redaction); behavioural defenses (ambiguity-loses, high-risk + underspecified ⇒ clarification, closed refusal vocabulary); defense delegation table.

  - **D4 — Performance contract (skinny):** budget shape per path (direct < 50 ms, lexical < 200 ms, RAG < 800 ms, reasoning < 3 s); cold start and reload budgets; backpressure rules (direct path always available; reasoning shed first); resource limits.
  - **D5 — Catalog distribution and trust:** signed bundles; AIOS root → publisher → bundle trust chain; hot-reload semantics with grace period; explicit operator-only rollback; out-of-scope items routed to L4, L10, and a future adapter-distribution sub-spec.

## DEC-006 — ProxGuard used as prototype donor and optional AIOS app

- **Context:** Local ProxGuard materials contain a working control-plane architecture pattern: manifest-driven simulation, deterministic policy, runtime adapters, release packaging, inbox handoff, production apply, audit events, DNS provider abstraction, and golden path tests. These patterns overlap with AIOS L3 Capability Runtime, L4 Policy Kernel, L8 Network, and L9 Evidence requirements.

ProxGuard Python code remains a prototype/reference artifact until tests prove donor behavior and AIOS-native ports or package wrappers are written.
