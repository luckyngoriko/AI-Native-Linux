## DEC-003 â€” Action Envelope + Lifecycle (S0.1) design choices

- **Context:** Rev.1 Â§13 sketched an action envelope and lifecycle but left major gaps: idempotency, causality, formal error model, lifecycle FSM precision, schema versioning, OpenTelemetry handling, sandbox profile binding, and dry-run modes. S0.1 closes these per scope option **B â€” Pragmatic+** (agreed at brainstorming).
- **Key decisions captured in [`XX_Cross_Cutting/01_action_envelope_lifecycle.md`](XX_Cross_Cutting/01_action_envelope_lifecycle.md):**
  - **Envelope style:** custom AIOS-native, proto-first, with `request`/`execution` field-level separation borrowed from Kubernetes API machinery. CloudEvents (event-style, not command-style) and full K8s API machinery (overkill for AIOS) were rejected.
  - **Top-level partition:** `schema_version`, `identity`, `request`, `execution`, `trace`. `identity` and `request` are caller-set and immutable after creation; `execution` is runtime-set and mutates over the lifecycle; `trace` is transport metadata.
  - **IDs:** ULID (Crockford base32, 26 chars), prefix-namespaced (`act_`, `intent_`, `plan_`, `corr_`, `evr_`, `polreq_`, `poldec_`, `appr_`). Chosen over UUID for chronological sortability and compactness.
  - **Idempotency:** Stripe-style separate `idempotency_key` field with 24h default TTL. Same key + same `hash(request)` deduplicates; same key + different hash returns `IdempotencyConflict`.
  - **Causality:** single-parent `parent_action_id` with cycle detection. Multi-parent saga deferred.
  - **Lifecycle:** five-phase FSM (`PENDING`, `RUNNING`, `SUCCEEDED`, `FAILED`, `ROLLED_BACK`) with K8s-style `conditions[]` for fine-grained facts. Six valid transitions; terminal phases are truly terminal (post-hoc rollback is a new envelope, not a transition).
  - **Result and Error:** two independent optional fields (not `oneof`), enabling `ROLLED_BACK` to populate both (cause + rollback summary). Population rules enforced by Capability Runtime.
  - **Error taxonomy:** ~30 canonical PascalCase codes grouped into validation/policy/auth/execution/verification/rollback/infrastructure. Recursive `cause` chain bounded at depth 8. `retryable` hint as a non-binding caller hint. English `message`; localization is a renderer concern.
  - **Versioning:** proto API style (`v1alpha1` â†’ `v1betaN` â†’ `v1` â†’ `v2`), not SemVer. Stable allows additive changes only; breaking changes require major version bump with 12-month dual-version support.
  - **Canonical encoding:** proto3 deterministic serialization for the wire and idempotency hashes; JCS (RFC 8785) for JSON projection in evidence. **BLAKE3** (256-bit) for all canonical hashes.
  - **Trace context:** standard W3C Trace Context, no AIOS-specific extensions; child actions inherit parent `trace_id`.
  - **Sandbox profile:** resolved at execution start as `max_restriction(policy_required, caller_request, adapter_default)`. Fail-closed if all unset.
  - **Dry-run modes:** `LIVE` (default), `VALIDATE` (schema/idempotency only), `SIMULATE` (full path with simulated execution; sandbox still applied; evidence marked `simulated=true`). Adapters that handle destructive operations must declare a `simulate` capability.
  - **gRPC interface:** simplified from rev.1's nine RPCs to six. Single entry point `SubmitAction` (replaces `Validate`/`EvaluatePolicy`/`Execute`/`Verify`/`Rollback` orchestration). Streaming `WatchAction` replaces polling. New `GetCapabilityRuntimeInfo` for version negotiation.
- **Out of scope (intentionally deferred):** subject canonical format (L4 identity), saga composition, approval delivery mechanics (L4), TTL/expiration, resource budgets.
- **Consequences:** L3 (Capability Runtime), L4 (Policy Kernel client), L5 (Cognitive Core), L9 (Evidence) now have a stable contract to build against. Phase 1 sub-specs (Capability Translator S1.1, Latency Tiering S1.2, AIOS-FS object model S1.3) can proceed without re-litigating action contract questions.
- **Status:** `CONTRACT` (design approved 2026-05-07; awaits implementation evidence to escalate to `REAL`).
- **Phase tag:** S0.1
- **Schema version:** `aios.action.v1alpha1`

## DEC-004 â€” Capability Translator as catalog-bound compiler, not command generator

- **Context:** Rev.1 required that AI never directly execute shell commands and instead produce typed actions. S0.1 defined the action envelope, but L5 still needed a precise contract for translating natural-language intent and planner steps into those envelopes at large catalog scale.
- **Decision:** Define the Capability Translator as a catalog-bound compiler. It may use LLMs for parsing, extraction, explanation, and candidate suggestion, but every READY result must validate against a known capability manifest, the selected target schema, and the S0.1 envelope schema. Vector retrieval is allowed for candidate discovery but cannot be final authority. Ambiguity returns clarification; missing schema fields block translation; unknown actions and free-form shell execution are rejected.
- **Consequences:** AIOS gets a scalable path for thousands of typed capabilities without becoming a prompt-to-shell system. L3 adapter manifests become the source of truth for action availability. L4 remains the policy authority. L9 receives translation evidence for auditability. S1.2 can now specify latency routing between direct translation and cognitive translation without redefining translator semantics.
- **Status:** `CONTRACT` draft (awaiting review before design approval)
- **Phase tag:** S1.1

- **Context:** Network is the single biggest exposure surface in AIOS. - **Decision:** Write the initial S8.1 contract (864 lines) defining the network policy enforcement plane: default-deny posture, per-subject outbound directives, exposure approval gates, AI external-call canonical pattern. - **Constitutional choices and why each:**
  - **AI subjects NEVER granted ALLOW_INTERNET / ALLOW_LIST_ONLY for arbitrary destinations.** Their network access flows only through L4.2 vault-brokered capability handles.   - **External-model-call canonical pattern.** AI calls to OpenAI/Anthropic/etc. flow as: typed `external_model_call` action â†’ policy decides â†’ vault broker holds the API key â†’ L8 brokers the connection on AI's behalf â†’ `EXTERNAL_MODEL_CALL_BROKERED` evidence. Any other path is a constitutional violation.
  - **Public exposure gated to recovery + STRONG approval + co-signer + FOREVER evidence + 4 h auto-termination + 5-minute heartbeat.** Public means routable internet; this is the highest-risk inbound class and gets the strongest discipline.
  - **Forbidden direct LANâ†’PUBLIC transition** enforced at L8 (must downgrade to LOOPBACK first). Already constitutional in S7.5; L8 enforces it again at the policy plane. Reason: the LOOPBACK ground state forces a deliberate constitutional re-acknowledgement before each escalation.
  - **Per-app signed `network_outbound_manifest`.** Every app/service/agent declares its outbound network footprint at install time; manifest is Ed25519-signed; modifications require re-issue + re-approval. Runtime breach (subject attempts a destination outside its manifest) refuses the connection with FOREVER `OUTBOUND_OUTSIDE_MANIFEST` evidence; repeated breach automatically degrades the subject's posture to AI_LOOPBACK_ONLY.
  - **Bounded HOST_FQDN fan-out (max 16 IPs).** Defense against DNS-based blast-radius expansion; an FQDN that resolves to > 16 IPs gets `ALLOWLIST_FQDN_FANOUT_EXCEEDED` and the entry is denied.
  - **DNS-over-TLS resolver allowlist constitutional.** Only AIOS-root-signed resolver list permitted; arbitrary DoT resolvers cannot be allowlisted (defense against malicious resolver substitution).
  - **PID â†’ subject_id correlator at the kernel boundary** (netlink/eBPF). The "who is this connection from?" question is answered deterministically at every connect() â€” defense against subject-id spoofing on connect.
  - **Mid-flight grant revocation < 250 ms.** When an OutboundDirective is revoked or an exposure is REVOKED, active connections are torn down within 250 ms; defense against long-lived backdoors after policy changes.
  - **Per-group network namespaces.** Linux network namespaces are partitioned per group with bridge to host loopback only â€” cross-group network access is impossible at the kernel level (binds INV-011).
- **Cross-spec follow-up queued:** 18 record types for S3.1 (7 FOREVER, 5 EXTENDED_60M, 6 STANDARD_24M); 3 S2.4 verification primitives (`network_subject_outbound_class`, `network_active_exposure_class`, `network_external_model_call_brokered_only`); 3 new S2.3 closed condition fields (`subject.network_outbound_directive`, `subject.ai_external_posture`, `target.exposure_class`); 1 candidate L0 invariant (`NETWORK_DEFAULT_DENY_OUTBOUND`) â€” promotion deferred to next L0 INV bundle revision.

## DEC-030 â€” S9.1 L1 Recovery Boundary (initial contract â€” closes the most-cited dangling reference in the AIOS constitution)

- **Context:** The recovery boundary is the most-cited concept in the AIOS constitution. INV-001 (recovery independent of L5), INV-004 (recovery boundary preserved), INV-012 (recovery required for system mutation), INV-022 (recovery aesthetically distinct), L4.1 hard-deny `RecoveryRequiredForSystemMutation`, S5.2 vault `RevealSecret` requires recovery_mode, S5.4 STRONG_SOLO override only-in-recovery, S7.1 surface composition recovery-only stack constraint, S7.5 web renderer RECOVERY exposure state, L4.3 `RECOVERY` SessionClass â€” every one cites recovery, but until now no contract specified what recovery actually IS.
- **Decision:** Write the initial S9.1 contract (735 lines) defining the recovery boundary in concrete filesystem and process terms. Proto IDL `aios.recovery.v1alpha1`. 8 closed enums: `RecoveryMode` (3 â€” NORMAL/RECOVERY/DEGRADED), `RecoveryEntryReason` (8), `RecoveryExitReason` (4), `RecoveryStage` (9-state linear FSM), `RecoveryMutableScope` (8), `RecoveryReadOnlyScope` (2), `RecoveryDeniedClass` (5), `RecoveryNetworkPosture` (3 â€” LOOPBACK_ONLY default / LAN_FOR_PROVISIONING / AIRGAP).
- **Constitutional choices and why each:**
  - **8-hour hard cap on recovery session.** Auto-reboot at TTL_EXPIRED. Cannot be loosened by any policy bundle. Reason: bounded blast radius â€” recovery is the highest-privilege mode in AIOS, so it must not be the default mode. A long-lived recovery session is a pre-attack staging surface.
  - **Exit-by-reboot only.** There is no in-place transition from RECOVERY to NORMAL. The only exit is reboot. Reasons: eliminates "did we really clean up state?" ambiguity; forces the kernel command line to change deterministically; prevents drift of recovery privileges into normal-mode operation.
  - **No-L5-in-recovery contract is mechanical, not aspirational.** L5 systemd units are masked in the recovery boot path. Any attempt to start L5 from the recovery shell fails closed with `L5StartProhibitedInRecovery` and emits FOREVER `RECOVERY_L5_START_BLOCKED` evidence. INV-001 made executable.
  - **Concrete mount discipline.** `/aios` is NOT MOUNTED in recovery. Group data (`/aios/groups/<group_id>/...`) is NOT MOUNTED writable; only read-only forensic attach is optional and emits FOREVER evidence. Recovery operators do not snoop on group content.
  - **Operator authentication via dedicated recovery prompt under `_recovery` KWin session.** No remote authentication in recovery. Hardware key preferred; passphrase fallback emits `HEAVY_AUTH_FALLBACK_USED`.
  - **Closed `RecoveryMutableScope` (8 paths).** Only POLICY_BUNDLE / CAPABILITY_CATALOG / VAULT_ROOT_MATERIAL / INVARIANT_BUNDLE / IDENTITY_BUNDLE / RECOVERY_OPERATOR_REGISTRATION / DEDICATED_KERNEL_PROMOTION / L1_BOOT_PARAMETERS are mutable in recovery. Even in recovery, the operator can only mutate these eight named scopes; everything else is read-only.
  - **Closed `RecoveryDeniedClass` (5 forbidden-even-in-recovery operations).** AI_AGENT_EXECUTION, WEB_PUBLIC_EXPOSURE, LAN_NETWORK_OPEN (default), THIRD_PARTY_BINARY_RUN (only L1 toolbox + signed AIOS binaries), EVIDENCE_LOG_REWRITE (binds S5.4 NonOverridableClass).
  - **Closed `RecoveryNetworkPosture`.** LOOPBACK_ONLY by default. LAN_FOR_PROVISIONING is operator-enabled, time-boxed, evidence-bound (every transition FOREVER). AIRGAP is for hardware repair. Recovery NEVER reaches public internet.
- **Cross-spec follow-up queued:** 10 FOREVER record types for S3.1 follow-up consolidation. The two remaining L1 sub-specs (`02_first_boot_flow.md` first-boot installer and `03_dedicated_kernel_pipeline.md` A/B kernel promotion) stay SHELL.
- **Consequences:** Every layer's "in recovery mode..." reference now has a concrete contract. The recovery boundary is mechanical: filesystem mount table + systemd unit masking + closed enum-gated mutation scopes + 8-hour TTL + exit-by-reboot. INV-001/004/012/022 are executable. L1 layer headline updated SHELL â†’ PARTIAL.

## DEC-029 â€” S5.4 L4 Emergency Override (initial contract â€” disjoint from approval; constitutional NonOverridableClass)

- **Context:** S2.3 Â§17 establishes hard-deny rules that cannot be loosened by any policy bundle. Approval (S5.3) only rescues `policy_pending` decisions â€” it cannot un-deny a hard-deny. The L4 overview (`00_overview.md` Â§planned_subspecs) reserved `05_emergency_override.md` (SHELL) for the only mechanism by which a hard-denied action may be executed: an explicit, costly, audited, multi-human-quorum Emergency Override. Without this contract, hard-deny is either too rigid (no escape hatch for legitimate recovery scenarios) or too leaky (override semantics are improvised).
- **Decision:** Write the initial S5.4 contract (701 lines) as a deliberately narrow, costly, and audited mechanism. Closed enums: `OverrideState` (7), `OverrideStrength` (3 â€” WEAK explicitly forbidden), `OverrideScope` (2 â€” no OPEN_SCOPE / ALL_ACTIONS / INDEFINITE), `OverrideTtlClass` (3 â€” no infinite TTL, no tier > 15 minutes), `OverrideDenialReason` (8), `OverrideRateClass` (3 â€” LIMITED_LOW â‰¤ 3/month default), `NonOverridableClass` (6 â€” `AIOS_RECOVERY_BREAK`, `EVIDENCE_LOG_REWRITE`, `VAULT_RAW_REVEAL_BYPASS`, `IDENTITY_KEY_FORGE`, `POLICY_DENY_LIST_DELETE`, `AIOS_CHROME_REPLACEMENT`).
- **Constitutional choices and why each:**
  - **Disjointness from approval.** Approval and override are two separate paths with disjoint preconditions: approval handles `policy_pending`, override handles `hard_deny`. Conflating them would erase the constitutional meaning of hard-deny rules.
  - **WEAK strength forbidden outside recovery.** Emergency Override is the most-trusted operation surface in the system; allowing solo-WEAK overrides outside recovery would let a single compromised session bypass any constitutional hard-deny. Default minimum is DUAL_HUMAN; STRONG_SOLO permitted only inside recovery boot (where physical access is already established).
  - **Channel separation invariant.** Each confirming subject's grant must arrive on a distinct ApprovalChannel (cite S5.3) â€” defends against single-machine compromise where one attacker drives both grants from the same session.
  - **NonOverridableClass.** Six classes are constitutionally hard-coded as un-rescuable by override: breaking the recovery boundary, rewriting evidence prior to current head, bypassing vault use-without-reveal for ON_REVEAL_ONLY material, forging identity keys, deleting policy deny rules, replacing AIOS chrome at the trust path. The only way to change these is rebuilding the system (out of runtime scope). This is what makes the constitution constitutional.
  - **No infinite TTL, hard cap 15 min.** Long-lived override grants are a pre-attack staging surface; capping at 15 min (and only that long in recovery) makes any override a tightly-scoped exception.
  - **Cooldown + monthly cap.** Per-subject cooldown (â‰¥ 5 min) and rate cap prevent override abuse via repeated micro-grants.
  - **Reversibility model.** A CONSUMED override cannot be unwound (effects are physical-world). The override RECORD is immutable FOREVER. Later overrides can change behavior going forward. Post-hoc forensic findings are recorded as separate `OVERRIDE_REVIEW` records â€” never as modifications. This is the cleanest possible audit invariant: every override stands on its own record.
- **Cross-spec follow-up queued:** eight FOREVER record types for S3.1 (`OVERRIDE_REQUESTED`, `OVERRIDE_QUORUM_RECEIVED`, `OVERRIDE_GRANTED`, `OVERRIDE_CONSUMED`, `OVERRIDE_DENIED`, `OVERRIDE_EXPIRED`, `OVERRIDE_REVOKED`, `OVERRIDE_REVIEW`); one new L7.2 NodeKind queued (`OVERRIDE_PROMPT` â€” visually distinct per L7.3 Î”E â‰¥ 25 HARMFUL state distinctness rule); fallback during the gap is APPROVAL_PROMPT NodeKind with `is_override = TRUE` ARIA-grade flag.
- **Consequences:** Hard-deny rules now have a precise escape hatch with full audit and constitutional bounds. The pattern of two disjoint rescue paths (approval for soft-deny, override for hard-deny) is established and locked. The L4 layer reaches `PARTIAL` headline with all five sub-specs `CONTRACT`-grade; promotion to `REAL` for the layer requires E2+ evidence (build/test) which is the implementation phase.
- **Status:** `REAL` (applied 2026-05-11).
- **Phase tag:** S5.4.
