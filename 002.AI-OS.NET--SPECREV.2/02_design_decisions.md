## DEC-001 — README "Self-Evolving Backend" reframed as "Adaptive Backend"

- **Context:** Rev.1 README contained a "Self-Evolving Backend" section that implied autonomous AI patching of backend code, in conflict with the spec discipline of typed actions, policy gating, and evidence-first execution. Rev.1 SPECIFICATION.md never sanctioned self-evolution.
- **Decision:** Reframe as "Adaptive Backend": AI may _propose_ backend patches, kernel adaptation adjustments, distro compatibility profiles, and new runtime adapters; production promotion always requires human approval. The Cognitive Core may not modify the Policy Kernel, the Evidence Log, the Vault Broker, or the recovery boot path through this pipeline.
- **Consequences:** README and SPECIFICATION are now consistent on the AI's bounded execution model. The pipeline (Observe → Propose → Sandbox simulation → Tests → Human review → Staged deployment → Monitor → Rollback) is fully explicit.
- **Status:** `REAL` (applied in initial commit `be318da`)
- **Phase tag:** S0.2

## DEC-002 — Repository layout: revision-per-folder, layer-per-subfolder

- **Context:** Rev.1 was a single `SPECIFICATION.md` file. As rev.2 grows, a single file becomes unwieldy. Comparable projects (NeuroCAD `000.000.Roadmap/NEUROCAD_UNIFIED_PLATFORM_SPEC_REV*`) use one folder per revision with topic-named files inside.
- **Decision:** For AIOS:
  - One folder per revision: `001.AI-OS.NET--SPECREV.1/`, `002.AI-OS.NET--SPECREV.2/`, …
  - Inside each rev.2+ folder, one folder per layer (L0–L10) plus a `XX_Cross_Cutting/` folder for contracts shared by multiple layers.
  - Inside each layer folder, files numbered `00_overview.md`, `01_<topic>.md`, `02_<topic>.md`, …
  - Rev.1 stays as a flat verbatim move (the original two files) since it pre-dates this convention.
- **Consequences:** Clean navigation, easy to grow (e.g., L4 will hold Policy Kernel, Vault Broker, and Identity Model as three separate sub-specs). Slightly more nesting than NeuroCAD's flat-files-per-revision approach, but justified by AIOS layers containing multiple distinct sub-systems each.
- **Status:** `REAL` (applied in this commit)
- **Phase tag:** infrastructure

## DEC-003 — Action Envelope + Lifecycle (S0.1) design choices

- **Context:** Rev.1 §13 sketched an action envelope and lifecycle but left major gaps: idempotency, causality, formal error model, lifecycle FSM precision, schema versioning, OpenTelemetry handling, sandbox profile binding, and dry-run modes. S0.1 closes these per scope option **B — Pragmatic+** (agreed at brainstorming).
- **Key decisions captured in [`XX_Cross_Cutting/01_action_envelope_lifecycle.md`](XX_Cross_Cutting/01_action_envelope_lifecycle.md):**
  - **Envelope style:** custom AIOS-native, proto-first, with `request`/`execution` field-level separation borrowed from Kubernetes API machinery. CloudEvents (event-style, not command-style) and full K8s API machinery (overkill for AIOS) were rejected.
  - **Top-level partition:** `schema_version`, `identity`, `request`, `execution`, `trace`. `identity` and `request` are caller-set and immutable after creation; `execution` is runtime-set and mutates over the lifecycle; `trace` is transport metadata.
  - **IDs:** ULID (Crockford base32, 26 chars), prefix-namespaced (`act_`, `intent_`, `plan_`, `corr_`, `evr_`, `polreq_`, `poldec_`, `appr_`). Chosen over UUID for chronological sortability and compactness.
  - **Idempotency:** Stripe-style separate `idempotency_key` field with 24h default TTL. Same key + same `hash(request)` deduplicates; same key + different hash returns `IdempotencyConflict`.
  - **Causality:** single-parent `parent_action_id` with cycle detection. Multi-parent saga deferred.
  - **Lifecycle:** five-phase FSM (`PENDING`, `RUNNING`, `SUCCEEDED`, `FAILED`, `ROLLED_BACK`) with K8s-style `conditions[]` for fine-grained facts. Six valid transitions; terminal phases are truly terminal (post-hoc rollback is a new envelope, not a transition).
  - **Result and Error:** two independent optional fields (not `oneof`), enabling `ROLLED_BACK` to populate both (cause + rollback summary). Population rules enforced by Capability Runtime.
  - **Error taxonomy:** ~30 canonical PascalCase codes grouped into validation/policy/auth/execution/verification/rollback/infrastructure. Recursive `cause` chain bounded at depth 8. `retryable` hint as a non-binding caller hint. English `message`; localization is a renderer concern.
  - **Versioning:** proto API style (`v1alpha1` → `v1betaN` → `v1` → `v2`), not SemVer. Stable allows additive changes only; breaking changes require major version bump with 12-month dual-version support.
  - **Canonical encoding:** proto3 deterministic serialization for the wire and idempotency hashes; JCS (RFC 8785) for JSON projection in evidence. **BLAKE3** (256-bit) for all canonical hashes.
  - **Trace context:** standard W3C Trace Context, no AIOS-specific extensions; child actions inherit parent `trace_id`.
  - **Sandbox profile:** resolved at execution start as `max_restriction(policy_required, caller_request, adapter_default)`. Fail-closed if all unset.
  - **Dry-run modes:** `LIVE` (default), `VALIDATE` (schema/idempotency only), `SIMULATE` (full path with simulated execution; sandbox still applied; evidence marked `simulated=true`). Adapters that handle destructive operations must declare a `simulate` capability.
  - **gRPC interface:** simplified from rev.1's nine RPCs to six. Single entry point `SubmitAction` (replaces `Validate`/`EvaluatePolicy`/`Execute`/`Verify`/`Rollback` orchestration). Streaming `WatchAction` replaces polling. New `GetCapabilityRuntimeInfo` for version negotiation.
- **Out of scope (intentionally deferred):** subject canonical format (L4 identity), saga composition, approval delivery mechanics (L4), TTL/expiration, resource budgets.
- **Consequences:** L3 (Capability Runtime), L4 (Policy Kernel client), L5 (Cognitive Core), L9 (Evidence) now have a stable contract to build against. Phase 1 sub-specs (Capability Translator S1.1, Latency Tiering S1.2, AIOS-FS object model S1.3) can proceed without re-litigating action contract questions.
- **Status:** `CONTRACT` (design approved 2026-05-07; awaits implementation evidence to escalate to `REAL`).
- **Phase tag:** S0.1
- **Schema version:** `aios.action.v1alpha1`

## DEC-004 — Capability Translator as catalog-bound compiler, not command generator

- **Context:** Rev.1 required that AI never directly execute shell commands and instead produce typed actions. S0.1 defined the action envelope, but L5 still needed a precise contract for translating natural-language intent and planner steps into those envelopes at large catalog scale.
- **Decision:** Define the Capability Translator as a catalog-bound compiler. It may use LLMs for parsing, extraction, explanation, and candidate suggestion, but every READY result must validate against a known capability manifest, the selected target schema, and the S0.1 envelope schema. Vector retrieval is allowed for candidate discovery but cannot be final authority. Ambiguity returns clarification; missing schema fields block translation; unknown actions and free-form shell execution are rejected.
- **Consequences:** AIOS gets a scalable path for thousands of typed capabilities without becoming a prompt-to-shell system. L3 adapter manifests become the source of truth for action availability. L4 remains the policy authority. L9 receives translation evidence for auditability. S1.2 can now specify latency routing between direct translation and cognitive translation without redefining translator semantics.
- **Status:** `CONTRACT` draft (awaiting review before design approval)
- **Phase tag:** S1.1

## DEC-014 — S4.1 AIOS-FS Namespace Layout (promotion from DRAFT NOTES to CONTRACT)

- **Context:** A namespace layout brainstorm was captured 2026-05-08 as `DRAFT NOTES` in `L2_AIOS_FS/05_namespace_layout.md`. The user approved Option A (system + groups split) and the five `§6` design defaults on 2026-05-09. Eight closed Phase 3 contracts (S0.1, S1.3, S2.1, S2.3, S2.4, S3.1, S3.2 plus unrefined L4/L5/L6) all reference "groups" and "users" implicitly; without a fixed namespace, every subsequent refinement would drift. The Sandbox Composer in particular compiles enforcement against group-scoped paths and required this contract to be implementer-grade before its boundary is touched.
- **Decision:** Promote `05_namespace_layout.md` to `CONTRACT` with the user-approved defaults locked in:
  - **Q1 → `groups/`** (not `tenants/`). `tenants/` deferred; not a breaking change to add later if needed.
  - **Q2 → flat group structure** with tag-based labels for pseudo-hierarchy. Subgroups deferred.
  - **Q3 → cross-group access forbidden by default** (option (c)). Capability-based (option (b)) deferred to a future Vault Broker spec; approval-based copy (option (a)) is still possible at action level (a copy is a typed action with policy approval, evidence trail, new version in destination — not silent sharing).
  - **Q4 → recovery-only system admin path.** No "settings UI for policy" in normal mode. Mutations to `/aios/system/policy/`, `/aios/system/capabilities/`, `/aios/system/vault/`, `/aios/system/recovery/` require recovery mode + human approver + `RECOVERY_EVENT` evidence (FOREVER retention).
  - **Q5 → both-tier inbox semantics.** Personal inbox at `/aios/groups/<g>/users/<u>/inbox/` shows only this user's pending items; group inbox at `/aios/groups/<g>/inbox/` shows all pending items in the group, filtered by privacy ceiling. Both are virtual S2.1 named views, never physical writable paths.

  Promotion deltas applied (12 bundles, equivalent to D1–D12 cycle pattern):
  - **D1 — Proto IDL + `NamespaceResolver` gRPC service:** `aios.namespace.v1alpha1` package; `ResolvePath`, `ListReservedNames`, `GetNamespaceInfo` RPCs. Full message types (`NamespacePath`, `NamespaceCatalogDescriptor`, `ResolutionError` with closed code enum). Appendix A holds the IDL.
  - **D2 — Closed reserved-name vocabularies at every scope:** `TopLevelReservedName` (2 entries), `SystemReservedName` (8 entries), `GroupReservedName` (11 entries including 3 virtual views), `UserReservedName` (8 entries including 2 virtual views). Adding a name is a versioned spec change.
  - **D3 — Closed identity regex:** `[a-z][a-z0-9_-]{0,62}` for group/user/agent/project IDs; reverse-DNS form for app IDs (max 5 dotted segments × 63 chars). Reserved `_*` prefixes (`_system`, `_recovery`, `_aios`, `_root`); `_system` is materialized at top level as `/aios/system/`, never addressable as a group_id.
  - **D4 — Deterministic stateless path resolution algorithm:** O(segments) parse with no backtracking; rejects path traversal (`..`/`.`/double slashes/non-ASCII), 32-segment depth cap, 4096-byte path length cap; produces typed `NamespacePath` with optional `(scope, group_id, user_id, reserved, subpath, is_virtual_view, namespace_catalog_version)`.
  - **D5 — Cross-group access default-deny constitutional invariant:** `CrossGroupAccessForbidden` hard-deny added to S2.3 (subject's primary_group_id ≠ target.group_id ⇒ DENY). Single exception: `_system` scope subject + recovery mode + `system_audit_read` capability + human approver. Cannot be loosened by policy bundle.
  - **D6 — Recovery-only system mutation boundary:** `RecoveryRequiredForSystemMutation` hard-deny added to S2.3 (mutations to `/aios/system/policy/`, `/aios/system/capabilities/`, `/aios/system/vault/`, `/aios/system/recovery/` require `is_recovery_mode = true` + human approver + `RECOVERY_EVENT` evidence). Cannot be loosened by policy bundle.
  - **D7 — AI subjects blocked from system admin:** `AISystemAdminBlocked` hard-deny added to S2.3, extending S2.3 §17 (AI subject + system-scope mutation ⇒ DENY even with `system_admin` capability). System admin is a human-only capability.
  - **D8 — Both-tier inbox semantics:** Personal inbox at user scope and group inbox at group scope; both virtual S2.1 named views with `ON_DEMAND` materialization, cardinality bound 10 000 items, cursor pagination. Mutation through inbox paths rejected with `VirtualPathNotWritable`. Outbox is also virtual.
  - **D9 — Namespace catalog versioning + signature:** Catalog descriptor JCS-canonical encoded; version `nscat_<hex_lower(BLAKE3(...))[:32]>` aligned with S0.1 §8.5. Ed25519 signature by AIOS root verified at load; signature failure puts resolver in degraded mode (resolves only the closed enum subset; refuses calls requiring reserved-id existence check). Resolved paths are stamped with the active catalog version; equality across versions returns false even if strings match (forces re-resolution after upgrade).
  - **D10 — Performance + adversarial robustness:** parse-only resolution p99 < 100 µs; with reserved-id existence check p99 < 1 ms; ASCII-only segments block homograph attacks; regex anchored and bounded (no catastrophic backtracking); pointer moves cannot cross scope (S1.3 CAS layer enforces); group registration is content-addressed (S1.3 optimistic concurrency resolves collisions).
  - **D11 — Cross-spec touch-up requirements explicitly enumerated:** S0.1 (target validation + `target.scope`/`group_id`/`user_id` populated), S1.3 (`ScopeBinding` field, no cross-scope pointer moves), S2.1 (new query fields, formal inbox/outbox views), S2.3 (three new constitutional hard-denies, five new condition fields including `subject.primary_group_id`), S2.4 (new primitive `aiosfs_path_in_namespace`, new property `NAMESPACE_NO_CROSS_GROUP_POINTERS`), S3.1 (records carry optional `(scope, group_id, user_id)` triple, two new record types `SYSTEM_ADMIN_OPERATION` STANDARD_24M and `CROSS_GROUP_ACCESS_DENIED` STANDARD_24M, query privacy ceiling extends to namespace scope), S3.2 (new `GROUP_FLOOR` composition source between policy_required and runtime_safety_floor, apply-time group ownership check). Plus L4 constraints (subject canonical id `<group_id>:<user_id>`, group as first-class identity, primary_group_id semantics), L5 constraints (agent objects with `group_owner` + nullable `user_owner`, capability bindings cannot escape scope), L6 constraints (app `installable_scope` field with three values `SYSTEM_ONLY`/`GROUP_ONLY`/`EITHER`, no dual-scope install of same app_id).
  - **D12 — Golden fixtures + telemetry + acceptance criteria + open deferrals:** six fixtures (personal household with family group, solo developer with separate personal+homelab groups, mixed work+personal with stricter work-finance group floor, path-traversal rejection cases, catalog version stamping, inbox virtual view filtering); ten metrics with bounded label cardinality (subject/group/user/action/profile/agent/app ids NEVER as labels — closed enums only); acceptance criteria as a checklist; open deferrals consolidated (cross-group capability-based sharing, federation policy, multi-tenant primary scope, subgroup hierarchy, per-group/per-user disk quotas, namespace replication across machines, read-only views of remote namespaces, catalog hot-reload mid-action, profile diffing UI for namespace audits).

- **Consequences:** S4.1 grows from 225 lines (DRAFT NOTES) to 939 lines (CONTRACT). Establishes the namespace as the single coherent projection through which L4, L5, L6, and all Phase 3 contracts reach the user. Default-deny cross-group access is now constitutional, not configurable. Recovery boundary from Rev.1 is preserved at the directory layer. Eight Phase 3 contracts now have an explicit, enumerated touch-up checklist for follow-up refinement cycles — those touch-ups are themselves new spec work, not in this commit. Cross-group sharing and federation become deliberate future work behind a Vault Broker contract; they cannot drift in silently.
- **Status:** `REAL` (applied 2026-05-09).
- **Phase tag:** S4.1 promotion.

## DEC-013 — S3.2 Sandbox Composition Language refinement (deltas D1–D12)

- **Context:** S3.2 landed as a 98-line draft in commit `dfa3be5`. Architecturally correct (most-restrictive composition, declarative profiles, fail-closed on missing enforcement, foreign apps wrapped) but missing typed surfaces, closed vocabularies, deterministic merge algorithm with winner trace, backend probe contract, lifecycle states, performance budgets, content-addressed identity, evidence integration, and adversarial robustness. The Sandbox Composer is the only component permitted to translate composed profiles into kernel-level enforcement; the contract needed to be implementer-grade before that boundary is touched.
- **Decision:** Apply twelve deltas without scope expansion:
  - **D1 — Proto IDL + `SandboxComposer` gRPC service:** `aios.sandbox.v1alpha1` package; `ComposeProfile`, `ValidateProfile`, `ApplyProfile`, `RevokeProfile`, `ExplainComposition`, `GetComposerInfo` RPCs; full message types (`SandboxProfile`, `CompositionInputs` with five typed sources, `CompositionMetadata` with per-field winner trace, `BackendCapabilities`, `CompatibilityProfile` with Wine/Waydroid/VM oneof, `CompositionError` with closed error code enum). Appendix has the IDL.
  - **D2 — Closed enum vocabulary:** `FilesystemMode` (READ_ONLY, READ_WRITE_PRIVATE, READ_ONLY_BROKERED, NO_ACCESS), `NetworkMode` (DENY_ALL, LOOPBACK_ONLY, HOST_LIMITED, EXPLICIT_ALLOWLIST, FULL — last requires policy override), `SecretsMode` (NO_SECRET_ACCESS, BROKER_ONLY, BROKER_PLUS_RECOVERY), `StreamCapture` (NONE, REDACTED, TRUNCATED, FULL), `LinuxCapability`, `CompatibilityKind`. Free-form strings forbidden in mode and enforcement choice fields.
  - **D3 — Deterministic five-source composition algorithm:** sources [adapter_default, app_manifest, user_request, policy_required, runtime_safety_floor] with most-restrictive-defaults composer source; per-field merge functions per type (boolean OR for restrictive, intersection for allow lists, union for deny lists, min for resource limits, strictness-ranked max for mode enums); floor enforcement after merge; policy invariant defense-in-depth check; complete `field_decisions` trace with winner source, winner value, and dropped candidate reasons.
  - **D4 — Backend capability detection + degraded mode:** `BackendCapabilities` snapshot via host probe (`/proc/self/ns`, `prctl(PR_GET_SECCOMP)`, `landlock_create_ruleset` ABI version, SELinux/AppArmor mode, cgroup version, nftables, XDG portals, Wine/Proton/Waydroid/KVM availability); required-backend mapping per profile field; fail closed unless `policy_required.allow_degraded = true` plus matching `approved_fallbacks`; `SANDBOX_DEGRADED_GRANTED` evidence with FOREVER retention; `host_capability_snapshot_id = hex_lower(BLAKE3(jcs(caps)))[:32]`.
  - **D5 — Compatibility runtime profiles:** typed `CompatibilityProfile.runtime` oneof (`WineProtonProfile`, `WaydroidProfile`, `VmFallbackProfile`); per-runtime safety floor (Wine: isolated prefix, `block_host_home = true` floor-enforced, portal file picker; Waydroid: per-app data, brokered clipboard, file portal only; VM fallback: explicit storage shares, evidence bridge mandatory, qemu-system invocation not virt-manager defaults); selection rejection rules (foreign binary with `kind = NATIVE_LINUX`, `kind = WINE_PROTON` without Wine, conflicting kinds across sources).
  - **D6 — Profile lifecycle + content-addressed identity:** `ProfileLifecycle` enum (DRAFT → VALIDATED → APPLIED → ENFORCING → REVOKED); `profile_id = "prof_" + hex_lower(BLAKE3(jcs_canonical(profile_without_metadata)))[:32]`; immutability post-apply (revoke + new profile to change enforcement); revocation creates new profile id, never mutates existing.
  - **D7 — Determinism contract:** `(adapter_default_id, manifest_hash, jcs_canonical(user_request), policy_decision_id, runtime_safety_floor_id, host_capability_snapshot_id, composer_version) → identical SandboxProfile`; cacheable per the same tuple; cache invalidated on host capability change, runtime floor bundle change, adapter default publication, or policy bundle version change; default cache TTL 5 minutes.
  - **D8 — Performance contract:** compose p95 < 5 ms fresh / < 1 ms cached; native apply p95 < 50 ms; Wine prefix bring-up p95 < 2 s; Waydroid container start p95 < 5 s; VM fallback boot p95 < 30 s; failure modes (`BackendUnavailable`, `CompositionConflict`, `ProfileApplicationFailed`, `ComposerInternal` — all fail closed); backpressure shedding via `COMPOSITION_BUDGET_EXCEEDED` when in-flight queue exceeds threshold.
  - **D9 — Evidence integration:** six new record types added to S3.1 closed `RecordType` vocabulary (`SANDBOX_PROFILE_COMPOSED` STANDARD_24M, `SANDBOX_PROFILE_APPLIED` STANDARD_24M, `SANDBOX_PROFILE_REVOKED` STANDARD_24M, `SANDBOX_DEGRADED_GRANTED` **FOREVER**, `SANDBOX_APPLICATION_FAILED` EXTENDED_60M, `SANDBOX_CAPABILITY_LIE` **FOREVER** dual with `TAMPER_DETECTED`); each record carries composition_id, profile_id, policy_decision_id, action_id, host_capability_snapshot_id; cross-spec dependency on S3.1 explicitly recorded.
  - **D10 — Adversarial robustness — omission and hints:** omitted fields resolve to most-restrictive defaults via `COMPOSER_DEFAULT` source (missing network → DENY_ALL, missing filesystem → NO_ACCESS, missing seccomp → `aios.deny-all` catalog id, missing secrets → NO_SECRET_ACCESS); `UserRequestHint` permissive aspects silently dropped with `dropped: true` and `drop_reason` recorded; reference cycles detected via DAG check at compose time and at composer startup; per-evaluation budget (≤ 100 fields, ≤ 1000 allowlist entries, ≤ 1000 denylist entries, ≤ 10 KB profile size); per-subject rate limits (10 000 compose / min, 1000 apply / min).
  - **D11 — Apply-time invariants + capability lie detection:** at apply, composer re-checks `profile_id` against content hash (no mid-flight tampering), re-probes host capabilities and compares against compose-time snapshot (mismatch → `HOST_CAPABILITY_LIE` plus `TAMPER_DETECTED` evidence with FOREVER retention), verifies target process exists, verifies action_id has `policy_decision` evidence with status ALLOW; violations emit `SANDBOX_APPLICATION_FAILED`.
  - **D12 — Golden fixtures + telemetry + cross-spec dependencies + open deferrals:** ten fixtures (basic 5-source merge, policy wins over permissive request, allow-list intersection, deny-list union, resource floor wins, required backend missing fails closed, degraded mode allowed by policy, Wine prefix wrapper composed correctly, circular reference rejected, apply-time capability lie detected); ten metrics with bounded label cardinality (`sandbox_compose_duration_seconds`, `sandbox_apply_duration_seconds`, `sandbox_degraded_mode_total` by missing_backend, `sandbox_capability_lie_total` by backend, etc. — subject/app/action/profile ids NEVER as labels); cross-spec table covers S0.1, S1.1, S1.2, S1.3, S2.1, S2.3, S2.4, S3.1 plus L4 vault and L4 identity; open deferrals consolidated (VM fallback orchestration to `03_compatibility_runtime.md`, compatibility knowledge DB to `05_compatibility_knowledge.md`, profile A/B testing, multi-tenant namespacing, hot-reload of safety floor, profile diffing UI, hardware-pinned secure enclaves).
- **Consequences:** S3.2 grows from 98 to 1107 lines. Becomes implementer-grade for the only component permitted to compile composed profiles into kernel enforcement. Closed enum vocabulary blocks free-form attack vectors. Five-source merge with per-field winner trace makes composition explainable and auditable.
