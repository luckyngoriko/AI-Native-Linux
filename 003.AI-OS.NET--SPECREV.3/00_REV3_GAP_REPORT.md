## 1. Executive verdict

Rev.3 is a **partial specification with a coherent thesis but an unfinished contract surface.** Of the five "materialized" contracts in `00_REV3_HOLISTIC_SPEC.md:61-69`, only S16 and S17 are decomposed into contract-grade sub-specs; S18/S19/S20 ship as single overview files that fold whole planes (closed enums, schemas, evidence vocabularies, acceptance criteria) into one document each. The **single biggest category of what's missing is evidence-and-traceability closure**: the ~52 new evidence record types declared across S18/S19/S20 (`S18 §15`, `S19 §12`, `S20 §16`) plus the ~12 native S16 types and all of S17's 31 capsule types are **completely absent from the Capella emitter manifest** — so the holistic spec's own acceptance criterion 7 (`§16`, "zero orphan INVs / zero cycles / zero inversions") passes only because the new Rev.3 vocabulary was never modeled. Compounding this, the security-audit's own P0 primitive (S16.4 Measured Boot) and four other recommended S16 sub-specs are named, depended upon, and gated against, yet were never written and are not even recorded as deferred. Finally, Rev.3 ships without the master index, decision log, and roadmap that Rev.2 carries, so the gap between "written" and "structurally complete" is itself untracked.

### T1-1 — S16.4 Measured Boot + Runtime Integrity (P0) is named, depended-upon, emits evidence, but does not exist `[CRITICAL]`

_Merges: STRUCT-01, STRUCT-S16-01, SCOPE-06 (S16.4 portion)._

- **What's missing.** A P0 sub-spec consolidating UEFI Secure Boot, kernel lockdown, signed-module enforcement, TPM 2.0 PCR set + signed-quote schema, IMA measurement/appraisal + EVM, dm-verity/IPE immutable root, per-boot root-integrity evidence, and the fleet attestation verifier. It is referenced once only — `00_PLANNING_NOTES.md:260` (P0, same priority as the materialized S16.1-3) — and never materialized.
- **Why it matters.** It is the high-assurance root of trust every other control attests against. `S16.1 01_…:147-148` emits `BOOT_INTEGRITY_POSTURE` and `CRYPTO_BOUNDARY_SELECTED` with **no schema**; `S16.3 03_…:77,:101` declares a "Boot posture" probe and P0 control `AIOS-SI-0001`; `S16.1 01_…:41-45` makes Secure Boot / lockdown / IMA-EVM / TPM **Required** for `STIG_ALIGNED` and `AIRGAP_HIGH`. The S16 overview `§4` sub-spec table (`00_overview.md:56-60`) lists only 01/02/03 with **no deferred row** — the gap is silently dropped, not declared.
- **What to write.** `S16_…/04_measured_boot_runtime_integrity.md`: closed `BootPosture`/`TPMQuote`/PCR-set schemas, IMA policy + appraisal lifecycle, lockdown-level enforcement, dm-verity/IPE root integrity, S8.5 firmware-drift coupling, fleet attestation verifier, dedicated evidence record types. ### T1-2 — Rev.3 evidence vocabulary (~95 record types) is structurally invisible to the traceability model `[CRITICAL]`

_Merges: TRACE-01, S17-02, STRUCT-S16-09, STRUCT-05/holistic, TRACE-05._

The Rev.3 "Produces" objects (`KernelCapabilityMatrix`, `DriverCapsule`, `NativeAISubject`, `PackagePassport`, `KernelBuildCandidate`, `AppCapsule`, …) appear in **no** manifest.

- **What to write.** Either (a) extend the S3.1 canonical enum (427→~520) with the new types + retention classes and re-run `extract/build`, or (b) if S3.1 is frozen, author a Rev.3 S3.1-delta evidence sub-spec and repoint `extract_record_types()` at it.
